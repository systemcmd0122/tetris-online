# テトリスバトル サイト・レビュー報告書

本リポジトリの全ファイルを確認し、技術的な問題点および改善の提案をまとめました。

---

### 1. セキュリティ (Security)
*   **【重大】管理者パスワードのハードコーディング (`admin.html`)**
    *   `ADMIN_PW = 'Runa1124'` がソースコード内に平文で記述されています。これはフロントエンド側で誰でも確認できてしまうため、管理画面の保護として機能していません。
    *   **改善案**: Firebase Auth の「カスタムクレーム（管理権限）」を導入するか、パスワード検証を Firebase Cloud Functions 等のサーバーサイドに移管してください。
*   **APIキー等の設定情報の重複**
    *   `FB_CONFIG` が `config.js` だけでなく `status.html` や `sprint-logic-main.js` にもハードコードされています。
    *   **改善案**: すべて `config.js` からインポートするように統一し、環境変数を利用してビルド時に注入する構成が理想的です。

### 2. アーキテクチャ (Architecture)
*   **ロジックの冗長性**
    *   `tetris-engine.js` という共通エンジンがあるにもかかわらず、`sprint-logic-main.js` で再度同じミノの定義やクラスが記述されています。
    *   **改善案**: すべてのモードで `tetris-engine.js` を唯一の正解（Single Source of Truth）として利用してください。
*   **ビジュアル要素のインライン化**
    *   背景のミノアニメーションやローディング画面が各 HTML にインラインスクリプトとして記述されています。
    *   **改善案**: `ui-effects.js` 等に抽出し、共通の CSS ファイルと共に管理することで、保守性とキャッシュ効率を向上させることができます。

### 3. UI/UX & モバイル対応
*   **モバイル対応の矛盾**
    *   `nav.js` の `Mobile Wall` 機能により、スマートフォンでの閲覧が制限されています。しかし、`tetris-multi.html` などにはタッチ操作用のコントローラーが実装されています。
    *   **改善案**: モバイル対応を正式にリリースするか、あるいは不要なモバイル用コードを整理するかの判断が必要です。実装が既にあるため、制限を解除しレスポンシブ対応を強化することを推奨します。

### 4. 保守性とパフォーマンス (Maintainability)
*   **ディレクトリ構造**
    *   ファイルがルート直下にフラットに置かれています。
    *   **改善案**: `/src`, `/assets`, `/styles` のように整理することで、開発時の見通しが良くなります。
*   **ビルドツールの未導入**
    *   CDN からの直接インポートは手軽ですが、バージョンの不整合（10.12.0 と 10.12.2 が混在など）や読み込み速度の低下を招きます。
    *   **改善案**: Vite 等のビルドツールを導入し、`npm` でパッケージを管理・ビルドすることで、堅牢なアプリケーションになります。

### 5. ロジック・バグの懸念
*   **ルームコード生成 (`multi-logic.js`)**
    *   `genCode()` が4桁の数字をランダムに生成していますが、重複（衝突）の確認が行われていないため、稀に既存のルームを上書きしたり参加できなかったりする可能性があります。
    *   **改善案**: 生成後に Firebase 上の存在を確認するか、より衝突耐性の高い英数字 ID を採用してください。

---

本サイトはゲームとしてのクオリティが非常に高く、Firebase のリアルタイム性も上手く活用されています。上記のエンジニアリング的な改善を行うことで、さらに信頼性の高いサービスになると確信しています。
